So instead HTML lets you attach a "shadow" DOM. Shadow DOM is a pradigm shift from what you might be used to. So there's a learning curve here. Quite a steep one depending on your expectations of how DOM works.

A shadow root is a little document within your page. ShadowRoots are encapsulated, so unless you have a reference to them you can't change them. Frameworks like React et al don't traverse into them so you're safe to do what you want in them.

While you don't have to use Shadow DOM, it's worth exploring because it's so powerful. It enables entirely new patterns in your application.
Shadow Roots are _how_ browsers work. A lot of built-in elements come with their own browser or "user agent" shadow root. For example an input element is literally just a handful of divs with some logic built in.

You can't access the user agent shadow DOM, it's what's called a "closed" shadowdom which is effectively fully private. Devtools exposes it so you can see what's going on. It's a cool way to learn about how elements work, you can see here all the pseudo elements are named. 

            <my-element>
              <template shadowrootmode=open>
                <button>Click me</button>
              </template>
            </my-element>
I want to clarify something here. DOM is what browsers use for UI. DOM has a standard serialisation format called HTML. I think this is important. DOM is to JS as HTML is to JSON. They're not 1 to 1 though, for example attributes can only be strings, HTML also has restrictions on what elements can be nested within others - for example in HTML you can't put a button in a button. In DOM you can, it has no such restriction.

In this example, to represent a shadow root in HTML, we need to use the template element with the shadowrootmode attribute. This is a new feature 2024 btw. When the parser encounters this tag it'll construct a shadow root for an element. The resulting DOM wont have the template element.
So if you ever open devtools and see a shadowroot template tag instead of the shadowroot, something went wrong.

Devtools inspects the live DOM, not the HTML.

Remember, this kind of idiosyncrasy has existed in HTML time immemorial, this type of discrepency is not unprecedented, but here it is perhaps the most evident.

However if you use react-dom renderToString or similar, where this would generate HTML from the server, and so this will work coming from the server.

In this way the framework has simply failed to abstract away the differences here. Worse still, React tries to suggest that HTML should serialise 1-1 with DOM, and if it misses attributes or elements it will issue a hydration warning.

Put simply, this is something frameworks need to address. If you're interested, React issue 26071 is around shadowdom and React.

Slots also have this concept of default content. Whatever you put inside the slot gets displayed if there's no light DOM children. So if a slot can't be filled, then the default is displayed.

If there's light dom nodes, i.e. some kind of direct children, then that overtakes the slot and the default content is hidden!

So here we have two elements, both have the same shadowdom but one has no light dom nodes, so the default content is displayed, while the other one has this text node which overrides the default.

Accessibility

I hope you've been following along so far because we're going to ramp it up a bit. You may have heard some folks espousing that "web components aren't accessible". As usual the truth is more nuanced than that. There are definitely problems, but there are also workarounds, and new proposals to solve these problems more easily. 

            <tab-nav>
              <template shadowrootmode="open">
                <slot role="tablist" name="tab"></slot>
              </template>
              <button role="tab">Tab 1</button>
              <div role="tabpanel">Panel 1</div>
            </tab-nav>
So a gotcha with Slots is that the slot element itslef hidden from the Accessibility Tree - it's children aren't, but the slot has no allowable role, which means HTML like this won't work. The role attribute works on almost every HTML element, the exceptions are mostly tags that live in the HEAD, like title, meta, script, etc.

So slots cannot have a role. There's also some cross browser issues around things like tabindex on slots. I think this is a bit of a problem, because it requires you to wrap the slot in more markup to make things work. I've raised issue 546 on html-aam about this - check the QR code. 

            <tab-nav>
              <template shadowrootmode="open">
                <div role="tablist">
                  <slot name="tab"></slot>
                </div>
              </template>
              <button role="tab">Tab 1</button>
              <div role="tabpanel">Panel 1</div>
            </tab-nav>
So the fix here would be to wrap the slot in a div element. Like I say this feels a less than ideal, so I'd like to see us fix this. I can't see a hugely practical reason why slots have this restriction. 

            <button aria-describedby="my-toolip">
            </button>
            <tool-tip id="my-tooltip">
              <template shadowrootmode="open">
                <slot id="description"></slot>
                <button aria-label="Close">X</button>
              </template>
              <p>Description for button</p>
            </tool-tip>
Another issue around shadowdom encapsulation is that Web Components can be tricky to get accessibility right with. Because shadowroots are encapsulated, their IDs are also encapsulated. For example if you had an interactive tooltip you wanted to describe a button, you cannot target IDs within the shadowdom, and so you have to resort to pointing to the whole element. This can be bad because in this example the description will also read the Close button label, when likely you just want the description.

This is less than ideal. There's a proposal to fix this, called Reference Target. This allows you to map certain attributes to certain parts of the shadowdom. So here for example we just want to say aria-describedby should only point to the slot, not the button. 

            <button aria-describedby="my-toolip"></button>
            <tool-tip id="my-tooltip">
              <template shadowrootmode="open"
                shadowrootreferencetargetmap="
                  aria-describedby: description
                ">
                <slot id="description"></slot>
                <button aria-label="Close">X</button>
              </template>
            </tool-tip>
So we can add this somewhat unweildy attribute called shadowrootreferencetargetmap, and this says "if something points to us as an aria-describedby reference, forward that down to this description ID instead.

I think this solves the problem quite well but it's still a proposal, and so today, this is still an unsolved issue. For the most part you can write accessible applications with web components, but something like this can be a real blocker to making your UI fully accessible and you'll sometimes need to work hard on finding a suitable workaround. 

            class ToolTip extends HTMLElement {
              connectedCallback() {
                this.ariaDescribedByElements = [
                  this.shadowRoot
                    .querySelector('#description')
                ]
              }
            }
Another potential solution to this is to leverage the new imperative ARIAMixin Properties. These properties let you assign various ARIA, such as describedby, using an imperative API which can allow you to more precisely control ARIA. By carefully using these properties you can achieve the same effect.

The problem here is that while a lot of the imperative properties like role, label, description are all available, the element properties still have limited availability. Only Safari has shipped at the moment. Chrome is close to shipping, Firefox is actively working on this. I anticipate that the story will be different soon. In the meantime you can use the non-element APIs but it will be a lot more work. As always test test test.

Okay let's do styling. The CSS issues are something people run into the most, and I think a lot of it is misunderstanding of how it works and how it works within the shadowdom.

Any style or link tag within a shadowroot only applies to that shadow DOM.

But styling Shadow DOM is a bit of a double edged sword, because none of the styles from outside apply to the interior ShadowDOM. This is, once again, a paradigm shift for teams. If you are using a CSS library, like a reset or a design system stylesheet, you'll need to add your CSS stylesheet to each shadowroot.

            class MyElement extends HTMLElement {
              connectedCallback() {

                  .adoptedStyleSheets.push(...document.styleSheets)
              }

One easy way to solve this could be to allow this pattern, where a ShadowRoot could simply adopt all of the stylesheets within the document. Sadly this isn't possible today because there's a difference between a constructed stylesheet, i.e. new CSSStyleSheet and one within a link or style tag. So this code would actually throw an error today. I think this distinction isn't really a technical barrier, I think it's solvable. I filed issue 10013 with the CSS working group to explore making this a reality and they've recently resolved to fixing this issue. 

            class MyElement extends HTMLElement {
              connectedCallback() {
                this.attachShadow({mode: 'open'})
                for (const { cssRules } of document.styleSheets) {
                  const sheet = new CSSStyleSheet();
                  for (const rule of cssRules) {
                    sheet.insertRule(rule.cssText);  
                  }
                  this.shadowRoot.adoptedStyleSheets.push(sheet)
                }
              }
            }
Meanwhile we can kind of fudge it with some gnarly code. It doesn't work perfectly, and there is a bit of a performance as it will serialize and re-parse all of the CSS in the document, but it kind of works... 

            <link rel=stylesheet href=my-design-system.css

One more way around this issue is to use a library. Brian Kardell made this neat library called half-light which lets you mark stylesheets as crossroot, and those stylesheets will be adopted into any open shadowroots without having to mess around too much further.

Ultimately though, my preferred solution is to lean into the shadow dom. Try to detatch from this notion of shared stylesheets and let each shadowroot control its own styles in a more fine grained way. There's lots of capabilities a ShadowRoot can offer if you let it. 

            <style>
              :root { --my-green: green; }

            <style>
              button, ::part(my-button) {
                background: green;
              }
            </style>
            <my-element>
              <template shadowrootmode=open>
                <button part="my-button">Click me</button>
              </template>
            </my-element>

              :state(busy) { filter: grayscale(1) }
              :state(busy)::part(my-button) { cursor: not-allowed }

              }
              set busy(value) {
                this.#i.states[value ? 'add' : 'delete']('busy')
              }
            }
You can also mint your own pseudo selectors with the colon-state selector. Again these can be styled globally, and can be combined with parts to create your own CSS API, which can become part of your glboal design sheet.

So in reality I think while there are restrictions around styling, it can be easily overcome. You need to bend it to you will a bit, but the net result can be something more delightful.

Takeaways

There are many more smaller issues. I could probably devote another talk to all the small issues. None of them are critical blockers though. I think this is all the ugly stuff. Like I said before no technology is perfect and no technology is salvation.

If it isn't evident already, I'm not here to sell you on anything. I've not really got any skin in the game. I'm not building a framework I want you to use, and I'm not trying to be a thought leader to sell some consulting course. I want you to pick sustainable technology choices and I want you to understand the problems that you might hit with them. I also want to recognise that the vested stakeholders like the web components community group are very aware of these issues, trying to fix them.

Should you use HTML Web Components?

So despite the issues, I still think this is a good technology. I framed at the beginning of my talk about how we need to detach from these siloed framework ecosystems, and I think HTML Web Components offer a real viable alternative. As you can see there's some paradigm shifts, but there's also some really cool features which enable new ways of thinking about this. I think they can compliment your existing codebases, without having to "buy in" to some shiny new tooling.
